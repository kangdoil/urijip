신혼부부 주거 조건 조율 서비스 — 배우자와 우선순위를 맞추고, 둘 다 만족하는 구역을 찾는다.

## 1. 문제 정의

신혼집을 구하는 부부는 각자의 우선순위가 다르지만, 이를 구조화해서 비교할 도구가 없다. 현재의 조율은 부동산 앱을 각자 보다가 대화로 절충하는 방식인데, 이 과정에서 세 가지 문제가 발생한다.

첫째, **우선순위가 암묵적이다.** "신축이 좋지"라는 말에서 그것이 포기 불가능한 조건인지 선호인지 구분되지 않아, 뒤늦게 갈등으로 드러난다. 둘째, **트레이드오프가 보이지 않는다.** "년식을 양보하면 무엇을 얻는지"를 수치로 확인할 수 없어, 조율이 감정 싸움이 된다. 셋째, **통근 균형을 계산하기 어렵다.** 두 직장에서 모두 적절한 거리인 구역을 찾는 것은 수작업으로는 사실상 불가능하다.

신혼부부는 이 문제가 가장 첨예한 세그먼트다. 대부분 맞벌이로 두 직장의 통근 균형이 실질 제약이고, 첫 공동 주거라 상대의 주거 우선순위를 서로 모르는 상태에서 시작하며, 결혼 일정이라는 명확한 데드라인이 있어 의사결정 동기가 강하다.

## 2. 가설

**H1 (가치 가설)** — 신혼집을 구하는 부부가 각자의 조건을 구조화해 입력하면, 시스템이 교집합 구역과 충돌 지점을 제시할 수 있고, 이것이 대화만으로 조율하는 것보다 후보 구역 결정을 빠르게 만든다.
검증: 결과 열람 세션 중 조율 루프(제안-동의)에 진입하는 비율, 세션당 후보 구역 저장 수.

**H2 (전파 가설)** — A(먼저 유입한 배우자)가 입력을 완료한 뒤 초대하는 구조("나는 다 했으니 너만 하면 결과가 나온다")는 B의 참여·완료 동기를 만들며, B 완료율이 서비스 성립의 병목이다.
검증: 초대 링크 클릭 → B 입력 시작 전환율, B 입력 시작 → 완료 전환율.

두 가설은 독립적으로 측정한다. H2가 실패하면 H1은 검증 기회 자체를 얻지 못하므로, 초기 개선 우선순위는 H2 퍼널에 둔다.

## 3. 타겟 사용자

행동 기준으로 정의한다 (태도 기준 배제).

**주 타겟 — 신혼부부**: 결혼 예정(1년 이내) 또는 혼인 신고 2년 이내이며, 첫 공동 거주 지역을 아직 확정하지 않았고, 최근 3개월 내 부동산 앱을 탐색한 2인.

참여자 역할: A = 세션 생성자(서비스 유입자), B = 초대 수신자(비유입자). B는 서비스를 검색한 적 없는 사람일 가능성이 높으므로, B의 온보딩은 A보다 마찰이 낮아야 한다 (가입 없음, 링크만으로 시작).

타겟 축소에 따른 이점: 상황 분기(커플/룸메이트/가족)가 사라져 온보딩이 한 단계 짧아지고, 카피·브랜딩을 신혼 맥락으로 고정할 수 있으며, v2에서 신혼부부 특례대출·특별공급 정보 연계라는 명확한 확장 경로가 생긴다.

## 4. 핵심 컨셉과 조율 모델

조율 방식은 **필수/선호 분리 필터링**을 채택한다 (가중치 병합 방식 기각 사유: 한쪽의 강한 조건이 평균에 희석됨).

조건 5개는 두 유형으로 나뉜다.

**상한 입력형 (항상 적용)** — 통근시간, 예산. 각자 상한 값을 입력하며 분류 대상이 아니다. 통근시간은 두 사람의 상한을 각각 적용하고, 예산은 낮은 쪽을 기본 적용하되 상한이 다르면 충돌로 표시한다.

**분류형 (필수/선호/무관)** — 평형, 년식, 인프라. 각자 3단계로 분류하며, 필수는 인당 최대 2개로 제한한다 (분류형 3개 전부를 필수로 지정하면 교집합 0건 위험이 커지므로. DB 트리거로 강제).

매칭 파이프라인:

1. **1차 필터 — 상한 + 필수 교집합.** 두 사람의 통근 상한과 합의 예산을 만족하고, 한쪽이라도 필수로 지정한 분류형 조건을 모두 충족하는 구역만 후보로 남긴다.
2. **2차 랭킹 — 선호 가중합.** 남은 후보를 A·B의 선호(nice) 충족 수로 점수화해 정렬한다.
3. **충돌 리포트.** 자동 절충하지 않는다. "A는 필수, B는 무관"인 조건과 상한이 다른 연속 조건(예산)을 명시하고, 각 조건을 완화했을 때 후보가 몇 곳 늘어나는지 수치로 제시한다. 결정은 두 사람이 한다.
4. **제안-동의 루프.** 기준 변경은 한쪽의 제안 → 상대의 동의를 거쳐야만 적용된다. 일방 변경 불가.

시스템의 역할 경계: 알고리즘은 트레이드오프 정보를 제공하고, 절충의 판단은 사용자에게 남긴다. "B가 양보하세요"류의 처방적 메시지는 금지한다.

### 5-1. 조건별 판정 기준 (구현 상세)

각 조건이 "충족/미충족"으로 판정되는 구체적 로직과 v1 구현 상태다. 기준값은 실 API 스펙 확인 후 정한 것으로, 세부 가중치·임계값 조정은 데이터 축적 후 재검토한다.

| 조건 | 판정 로직 | 데이터 소스 | 구현 상태 |
|---|---|---|---|
| 통근시간 | 참여자별 거점→구역 자동차 소요시간 ≤ 개인 상한. 출발·도착지가 5m 이내라 카카오가 경로 계산을 거부하는 구간(code 104)은 도보 수준인 1분으로 대체 | 카카오모빌리티 자동차 길찾기 API. `(출발 100m 격자, 구역, 수단)` 단위로 `commute_cache` 선조회 후 미스만 호출 | 구현 완료 — `src/lib/commute.ts`. 원래 ODsay(대중교통)로 구현했으나 API 키가 IP/URI 등록 방식이라 배포·개발 환경마다 인증이 막혀(`ApiKeyAuthFailed`) 카카오(자동차)로 전환 |
| 예산 | 구역 최근 6개월 실거래 평균가 ≤ 합의 상한(두 사람 중 낮은 쪽) | 국토부 아파트매매 실거래자료 API → `area_stats.avg_price_krw` | 구현 완료 — `scripts/refresh-trade-stats.ts` |
| 평형 | 구역 내 최근 6개월 거래 중 전용 59㎡ 이상 비중이 과반(50%) 이상이면 충족 | 국토부 아파트매매 실거래자료 API (같은 배치가 예산과 함께 처리) | 구현 완료 — `area_stats.size_59_ok` |
| 년식 | `현재연도 - built_year_avg ≤ 10` 이면 "신축" 충족 | 국토부 아파트매매 실거래자료 API의 건축년도(buildYear) → `area_stats.built_year_avg`. 별도 건축물대장 API는 지번 단위 개별 조회라 구역 전체 집계에 비효율적이라 채택하지 않음 | 구현 완료 |
| 인프라 | 마트·병원·공원 3항목 중 2개 이상 충족 시 인프라 충족 (§5 본문 기준) | 상가정보 + 심평원 + 도시공원 API → `area_stats.{mart,hospital,park}_ok` | 구현 완료 — `scripts/refresh-{mart,hospital,park}-stats.ts` |

예산·평형·년식 데이터 해상도 한계: 실거래가 API는 법정동 단위까지만 구분되고 행정동 단위가 아니다. "정자1동/정자2동/정자3동" 같은 행정동은 모두 법정동 "정자동"의 거래로 근사하며(행정동명 끝 숫자 제거로 대응 법정동명 추정), 해당 법정동에 거래가 없으면 시군구 전체 평균으로 대체한다 (초기 배치 기준 119개 구역 중 29개가 이 대체를 탔다).

인프라 3항목 세부 기준 (모두 구역 대표좌표 기준 직선거리 근사 — 통근시간 산정에 카카오 자동차 라우팅을 붙인 뒤에도 인프라 3항목은 아직 직선거리 근사로 남아있다. 실제 차량 라우팅으로 교체하면 정확도가 오를 여지가 있어 v2 후보로 남겨둔다):

- **마트** (`mart_ok`): 반경 3km 이내에 상권업종 소분류 "슈퍼마켓"(indsSclsCd=`G20404`, 대형마트 포함 — 이 데이터셋엔 대형마트 전용 소분류가 없어 슈퍼마켓 소분류 전체를 인정)이 1곳 이상. 소상공인시장진흥공단 상가(상권)정보 반경조회 API(`storeListInRadius`). 초기 배치 기준 대상 지역 전체(119/119)가 충족으로 나올 만큼 판정력이 약해 추가 조정 여지가 있다
- **병원** (`hospital_ok`): 반경 10km 이내에 종합병원(`clCd=11`) 또는 상급종합병원(`clCd=01`, 종합병원 요건을 상회하는 상위 등급이라 함께 인정)이 1곳 이상. 건강보험심사평가원 병원정보서비스 API(`getHospBasisList`)
- **공원** (`park_ok`): 반경 800m(도보 10분, 분속 80m 가정) 이내에 도시공원이 1곳 이상. 전국도시공원정보표준데이터 API

데이터 갱신은 `npm run refresh:{park,mart,hospital,trade}-stats`로 수동 실행하며, 자동 스케줄링은 v1 범위 밖이다.

반경 값(마트 3km/병원 10km)은 애초 평균 차량속도 30km/h 가정으로 각각 5km/20km였으나, 마트 기준은 "슈퍼마켓" 소분류가 동네 슈퍼까지 넓게 잡혀 초기 배치에서 119개 구역 전부가 충족으로 나올 만큼 판정력이 없어 3km로 좁혔다. 데이터 갱신은 `npm run refresh:{mart,hospital,park}-stats`로 수동 실행하며, 자동 스케줄링은 v1 범위 밖이다.

## 6. 사용자 플로우와 화면

플로우: 거점·통근 상한 → 예산 상한 → 조건 분류 → 초대/대기 → 결과 → 조율 (6화면). 상황 선택 화면은 신혼부부 단일 타겟 확정으로 삭제.

| 화면 | 핵심 설계 결정 |
|---|---|
| ① 거점·통근 | "직장"이 아닌 "자주 가는 곳"(직장/학교/부모님 집/직접 입력). 통근 상한 슬라이더가 지도 도달 범위와 연동. "문에서 문까지, 도보·환승 대기 포함" 기준 명시 |
| ② 예산 상한 | 슬라이더 입력. 두 사람의 상한이 다르면 낮은 쪽을 기본 적용하고 충돌로 표시 |
| ③ 조건 분류 | 분류형 3개(평형·년식·인프라)를 탭 방식 카드 소팅. 필수 슬롯 2개를 상시 노출해 "아껴 쓰는 자원"으로 인식시킴. 분류 완료 조건은 하단 칩으로 누적, 재분류 가능 |
| ④ 초대/대기 | A 입력 완료 후 초대 (빈손 초대 대비 B 완료율 우위 가정). 카카오톡 공유 + 동적 OG 태그 ("OO님이 신혼집 찾기에 초대했어요"). "상대 입력 완료 전 내 조건 비공개" 명시 (입력 편향 방지, RLS로 강제) |
| ⑤ 결과 | 헤드라인이 추천 근거를 먼저 제시. 구역 카드마다 A/B 통근시간 병기. 충돌 리포트에 완화 시 후보 증가량 수치 제시. 결과 공유 진입점 배치 |
| ⑥ 조율 | 조건 조정 시 후보 수 실시간 반영. 이산 충돌은 양자택일 버튼, 연속 충돌(예산)은 두 사람 값을 양 끝으로 하는 슬라이더. 변경은 제안 → 상대 동의로만 적용 |

**결과 공유 (v1 포함)**: 결과 화면에서 추천 구역 요약을 이미지 카드로 생성해 카카오톡·인스타그램으로 공유한다. 카드에는 구역명, 매칭률, 두 사람의 통근시간을 담되 예산 등 민감 정보는 기본 제외한다 (포함 여부 토글 제공). 공유 카드에 서비스 유입 링크를 포함해 신규 A 획득 채널을 겸한다 — 신혼부부는 준비 과정을 지인·SNS에 공유하는 경향이 강해, 결과 공유가 자연스러운 획득 루프가 될 수 있다.

## 7. 지표 체계

| 구분 | 지표 | 정의 |
|---|---|---|
| North Star | 주간 공동 결과 도달 세션 수 | 두 참여자가 모두 입력을 완료하고 결과 화면을 각자 1회 이상 열람한 세션 |
| KPI① | B 초대 수락률 | 초대 링크 클릭 → B 입력 시작 |
| KPI② | B 입력 완료율 | B 입력 시작 → 조건 분류 완료 |
| KPI③ | 조율 루프 진입률 | 충돌이 있었던 결과 열람 세션 중 제안 1건 이상 발생 비율 |
| 보조 | 결과 공유율 / 공유 경유 신규 세션 | 결과 도달 세션 중 공유 발생 비율, 공유 링크 경유 세션 생성 수 |
| 가드레일① | 필수 교집합 0건 비율 | 완결 세션 중 후보 0곳 비율. 상승 시 조건 설계·필수 제한 재검토 신호 |
| 가드레일② | A 온보딩 이탈률 | 거점 입력 진입 → 초대 화면 도달 실패 비율 |

주의: KPI③은 H1의 대리 지표이며, 분모를 "충돌이 있었던 세션"으로 한정한다. 충돌이 없는 세션의 조율 미진입을 "가치 없음"으로 오독하지 않기 위함이다.

이벤트 트래킹은 Mixpanel, 세션 단위 funnel: `session_created → invite_sent → invite_opened → b_started → b_completed → result_viewed_a → result_viewed_b → result_shared → proposal_created → proposal_accepted`.

## 8. MVP 범위

**포함**: 6화면 전체 플로우, 상한+필수 필터링·선호 랭킹 엔진, 통근시간 계산(자동차), 충돌 리포트, 제안-동의 조율 루프, 카카오톡 초대, 결과 이미지 카드 공유, 구역 단위 추천 (행정동 프리셋).

**초기 지원 지역 (확정)**: 강남·판교 두 업무지구의 통근권이 겹치는 경기 동남부 — 성남시, 하남시, 과천시, 의왕시, 용인시(수지구·기흥구), 광주시를 1차 범위로 한다 (행정동 119개). 선정 근거: ① "한 명은 강남, 한 명은 판교" 조합이 흔해 통근 균형이라는 핵심 기능이 가장 잘 작동하는 무대, ② 서울 대비 신혼부부 예산과 맞는 진입 가격대의 구역이 충분, ③ 행정동 규모가 통근 캐시·데이터 배치 비용으로 감당 가능. 지역 확장은 areas 테이블 시드 추가만으로 가능하도록 지역 하드코딩을 금지한다.

**1차 확장 (배포 전 물량 확보)**: 수원시(영통구·권선구), 화성시(동탄구)를 추가해 119 → 152개 행정동으로 넓혔다. 원래 계획한 확장 기준(NSM 안정화 + 가드레일① 임계 이하 유지 후 확장)은 아직 충족되지 않았지만, 초기 배포 시점에 후보지가 너무 적다는 판단으로 선반영했다 — 두 지역 모두 신분당선(광교)·SRT·GTX-A로 강남·판교 통근권에 들어와 "통근 균형" 핵심 가치는 유지된다. **구역 단위는 시군구가 아닌 행정동을 그대로 유지한다** — 시군구 단위로 넓히면 통근시간 정밀도가 무너져(같은 시 안에서도 위치별 통근시간 차이가 30분 이상 날 수 있음) 이미 부동산 앱에서 알 수 있는 수준의 조악한 추천이 되므로 채택하지 않았다. 다음 확장은 인접 통근권(분당선·신분당선·경강선 연장 축) 순으로, 이번엔 원래 기준대로 NSM·가드레일① 신호를 보고 판단한다.

데이터 해상도 참고: 화성 동탄구는 동탄1~9동이라는 행정동명이 실제 법정동명(반송동·석우동 등 옛 지명)과 달라 §5-1의 법정동 매칭 근사가 통하지 않고, 9개 동 전부 시군구 평균으로 대체됐다. 예산·년식·평형 정밀도가 이 지역만 상대적으로 낮다는 뜻이라 v2에서 법정동 매핑을 보강할 필요가 있다.

**제외 (v1 아웃오브스코프)**:
- 매물 단위 추천 — 구역까지만 추천하고 매물 탐색은 기존 부동산 앱에 위임. 매물 데이터 크롤링은 ToS 위반으로 채택 불가 (가는 집에서 확인된 제약)
- 대중교통 통근시간 — ODsay API 키 인증 문제로 자동차만 v1 제공 (§9 리스크 참고). 대중교통 이용자에게는 통근시간이 부정확할 수 있다는 한계가 있음
- 신혼부부 특례대출·특별공급 정보 연계 (v2 확장 경로)
- 인프라 세부 가중치 커스터마이즈
- 계정 시스템 고도화 — B는 익명 인증으로 시작, 계정 전환은 v2

**폴백 요구사항**: 필수 교집합이 0건일 때 "A의 필수만 반영한 후보"와 "B의 필수만 반영한 후보"를 나란히 제시한다. 빈 결과 화면은 세션 폐기로 직결되므로 v1 필수 기능이다.

**피드백 수집 (v1 포함)**: 결과 화면 하단에 낮은 강도의 배너/시트로 노출한다. 1단계로 "이 결과가 도움이 됐나요?" 1탭 반응(👍/👎)을 받고, 👎 선택 시에만 짧은 서술형 입력을 이어서 받는 2단계 구조로 한다.

트리거 조건: 주 트리거는 `result_viewed_b` 이후 결과 화면 체류(스크롤 또는 일정 시간)가 확인된 시점, 보조 트리거는 `proposal_accepted` 시점이다. 둘 중 먼저 도달하는 시점에 노출하고, 세션당 1회만 노출한다 (재노출 금지).

배제 원칙: 온보딩 전 화면(거점·예산·조건 분류)과 B의 초기 진입 직후에는 절대 노출하지 않는다. B 퍼널이 가드레일②로 관리되는 최대 리스크이므로, 피드백 수집이 이탈 요인이 되어서는 안 된다.

## 10. 리스크와 열린 질문

| 리스크 | 내용 | 대응 |
|---|---|---|
| B 퍼널 붕괴 | B가 링크를 열지 않거나 중도 이탈하면 서비스가 성립하지 않음 | H2 퍼널을 최우선 계측. 초대 메시지 문구·OG 이미지 실험 |
| 교집합 0건 빈발 | 상한 2개 + 필수 조건 조합이 배타적이면 빈 결과 반복 | 필수 2개 제한 + 폴백 화면. 가드레일① 모니터링 |
| 통근 API 인증/쿼터 | (발생함) ODsay는 API 키를 IP 또는 URI(Service URI)에 고정하는 방식이라, 등록해둔 값과 실제 호출 환경(로컬/샌드박스/배포)이 다르면 매 요청이 `ApiKeyAuthFailed`로 막혔다 — Vercel처럼 아웃바운드 IP가 고정되지 않는 배포 환경과는 구조적으로 안 맞음. 카카오모빌리티로 전환해 해결했으나 대신 대중교통 대신 자동차 기준이 됐고, 카카오도 일일 무료 호출 10,000건 한도가 있음 | 격자 캐시 + 행정동 프리셋으로 호출량 자체를 줄임. 쿼터 초과 시 대안 API(예: T맵 모빌리티) 조사 필요 |
| 데이터 신선도 | 실거래가 평균이 시장 급변기에 왜곡 | 최근 6개월 집계 + 갱신 시점 표기 |
| 조건 축소의 부작용 | 조건 5개가 신혼부부 의사결정을 충분히 커버하지 못할 가능성 (예: 역세권을 통근시간이 대체하지 못하는 케이스) | 결과 화면에 "빠진 조건이 있나요?" 피드백 수집, 조건 추가는 데이터 기반으로 결정 |

열린 질문: ① 공유 카드의 민감 정보(예산) 기본 노출 정책 최종 확정.

**의도적 보류 사항**: 저장 구역 시세 변동 알림 등 재방문 유도 기능은 v1에서 의도적으로 제외한다. 사용자가 실제로 무엇을 저장·추적하고 싶어 하는지에 대한 데이터가 아직 없는 상태에서 기능을 확정하면 추측 기반 설계가 되므로, 우선 피드백 수집(8장)과 기본 지표로 재방문 동기를 관찰한 뒤 방향을 정한다.
