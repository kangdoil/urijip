# 경기 서북부 지역 확장 설계

## 배경 및 목적

현재 서비스는 경기 동남부(성남·하남·과천·의왕·용인 수지/기흥·광주) + 1차 확장(수원 영통/권선, 화성 동탄), 총 152개 행정동만 지원한다. PRD(`docs/PRD-우리집_v2.md` §8)는 이 범위를 "강남·판교 통근권"으로 규정하고, 확장 기준을 "인접 통근권(분당선·신분당선·경강선 축)부터"로 명시하고 있다.

이번 확장 대상인 고양(일산동구·일산서구)·파주·김포·양주·의정부는 강남/판교 통근권이 아니라 **서울도심·여의도·GTX-A 축**에 가까운 별개 권역이다. PRD의 확장 기준 문구와 결이 다르므로, 데이터 추가와 함께 PRD §8도 갱신한다.

## 아키텍처: 코드 변경 불필요

`areas` 테이블은 지역 제약이 없는 범용 스키마(`code, name, sigungu, lat, lng`)이고, 통근시간 계산(`src/lib/commute.ts`)은 `areas` 테이블 전체를 좌표 기준으로 조회해 캐시를 채우는 구조다(`ensureCommuteForOrigin`이 지역 필터 없이 전체 스캔). 사용자가 지역을 직접 고르는 화면도 없다 — 온보딩에서 입력한 거점 좌표와 통근 상한만으로 자동 후보/제외된다.

따라서 이번 확장은 **데이터 추가만으로 완결**된다:
1. `data/areas-seed.json`에 신규 50개 행정동 추가
2. `npx tsx scripts/seed-areas.ts` 재실행 → Supabase `areas` 테이블 upsert
3. `scripts/seed-areas.ts` 상단 주석에 확장 이력 한 줄 추가 (기존 확장 이력 기록 관례)
4. `docs/PRD-우리집_v2.md` §8 문구 갱신

코드 로직(매칭 파이프라인, 통근 캐시, RLS, 트리거) 변경은 없다.

## 대상 행정동 (50개, 주거 밀집 핵심동만 선별)

기존 원칙(경기동남부는 시/구 전체, 수원/화성은 특정 구 전체)과 달리, 이번엔 사용자 요청에 따라 **개별 행정동 단위로 주거 밀집 지역만 선별**했다. 조사 과정에서 존재하지 않는 행정동(옥정3동, 민락1동, 김포 감정동/걸포동 — 모두 법정동이거나 실재하지 않음)이 정정되었다.

### 데이터 출처 및 방법

- **좌표**: 각 행정동 행정복지센터(주민센터) 주소를 조사한 뒤 Nominatim(OSM) API로 지오코딩. 기존 119개 항목과 동일한 방식.
- **행정동 코드**: `code.go.kr`(행정표준코드관리시스템)이 JS 검색폼 기반이라 자동 조회가 불가능해, 이번엔 오픈소스 데이터셋(`vuski/admdongkor` ver20260701, `KIKcd_H` 2023-05-01 스냅샷)의 10자리 행정기관코드를 대조 사용했다. **이는 기존 두 방식(행안부 공식 조회 / SBIZ 상가정보 API)과 다른 세 번째 소스**이며, 알려진 코드 사례와 구조적 일치는 확인했으나 공식 API로 전수 재검증하지는 못했다.
- **좌표/코드 추정치 처리**: 사용자 결정에 따라 아래 8개 추정치 항목도 **그대로 포함해서 진행**한다(기존 33개 확장분에도 "추정치" 표시 관례가 있음). 각 항목의 `source_note`에 추정 근거와 재검증 필요 여부를 남긴다.

**추정치/주의 항목 (8건)**:
- 중산2동 — 임시청사(풍산프라자) 좌표
- 탄현2동 — 임시청사(이원마트) 좌표, 2029년 신청사 예정
- 옥정2동 — 2023년 분동 후 임시청사 추정
- 회천1동 — 주소 상충(시청 공식 vs 일부 뉴스 신청사 주소), 공식 페이지 값 채택
- 송산3동 — 지번 매칭 실패로 인근 버스정류장 좌표 대체
- 운정3동 — 도로명주소 미등록으로 법정동(야당동) 중심좌표 대체
- 운정5동 — 도로명주소 미등록으로 법정동(목동동) 중심좌표 대체
- 마산동 — 주소 상충(김포한강8로 246 vs 김포한강3로 432), POI 이름 검색으로 대체 확인

1. `npx tsx scripts/seed-areas.ts` 실행 후 콘솔 출력으로 upsert 건수(202건) 확인.
2. Supabase에서 `select count(*) from areas` 로 실제 행 수 확인.
3. 신규 지역(예: 고양시 좌표)을 거점으로 온보딩을 진행해 `/api/odsay/batch-commute` 호출 후 `commute_cache`에 해당 `area_code`들의 행이 정상 생성되는지 확인.
4. 결과 화면에서 신규 지역이 `sigungu-filter-sheet`와 지도(`result-map-sheet`)에 정상적으로 그룹핑/표시되는지 확인.
