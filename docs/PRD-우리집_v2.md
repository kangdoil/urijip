# PRD · 우리집

신혼부부 주거 조건 조율 서비스 — 배우자와 우선순위를 맞추고, 둘 다 만족하는 구역을 찾는다.

| 항목 | 내용 |
|---|---|
| 문서 버전 | v0.8 |
| 작성일 | 2026-07-12 |
| 상태 | NSM 교체(확정 클릭률), 확정-이미지 동시 저장, 이벤트 병합 반영 |
| 관련 프로젝트 | 가는 집(1인용 비교·제외 서비스)과 별도 서비스, 조건 프레임워크 일부 공유 |

---

## 1. 문제 정의

신혼집을 구하는 부부는 각자의 우선순위가 다르지만, 이를 구조화해서 비교할 도구가 없다. 현재의 조율은 부동산 앱을 각자 보다가 대화로 절충하는 방식인데, 이 과정에서 세 가지 문제가 발생한다.

첫째, **우선순위가 암묵적이다.** "신축이 좋지"라는 말에서 그것이 포기 불가능한 조건인지 선호인지 구분되지 않아, 뒤늦게 갈등으로 드러난다. 둘째, **트레이드오프가 보이지 않는다.** "년식을 양보하면 무엇을 얻는지"를 수치로 확인할 수 없어, 조율이 감정 싸움이 된다. 셋째, **통근 균형을 계산하기 어렵다.** 두 직장에서 모두 적절한 거리인 구역을 찾는 것은 수작업으로는 사실상 불가능하다.

신혼부부는 이 문제가 가장 첨예한 세그먼트다. 대부분 맞벌이로 두 직장의 통근 균형이 실질 제약이고, 첫 공동 주거라 상대의 주거 우선순위를 서로 모르는 상태에서 시작하며, 결혼 일정이라는 명확한 데드라인이 있어 의사결정 동기가 강하다.

## 2. 가설

**H1 (가치 가설)** — 구조화된 조건 입력과 트레이드오프 정보 제시는, 신혼집을 구하는 두 사람이 공동 후보 구역을 확정하게 만든다.
검증: 공동 후보 확정률(행동 신호) + 결정 자기보고(선언 신호)의 교차 측정. 상세 정의는 7장.
주: 이전 버전의 "대화만으로 조율하는 것보다 빠르게"는 대조군이 없어 제품 내에서 반증 불가능하므로, 사용자 상태 변화(미결정 → 공동 확정)로 재정의했다.

**H2 (전파 가설)** — A(먼저 유입한 배우자)가 입력을 완료한 뒤 초대하는 구조("나는 다 했으니 너만 하면 결과가 나온다")는 B의 참여·완료 동기를 만들며, B 완료율이 서비스 성립의 병목이다.
검증: 초대 링크 클릭 → B 입력 시작 전환율, B 입력 시작 → 완료 전환율.

두 가설은 독립적으로 측정한다. H2가 실패하면 H1은 검증 기회 자체를 얻지 못하므로, 초기 개선 우선순위는 H2 퍼널에 둔다.

## 3. 타겟 사용자

행동 기준으로 정의한다 (태도 기준 배제).

**주 타겟 — 신혼부부**: 결혼 예정(1년 이내) 또는 혼인 신고 2년 이내이며, 첫 공동 거주 지역을 아직 확정하지 않았고, 최근 3개월 내 부동산 앱을 탐색한 2인.

참여자 역할: A = 세션 생성자(서비스 유입자), B = 초대 수신자(비유입자). B는 서비스를 검색한 적 없는 사람일 가능성이 높으므로, B의 온보딩은 A보다 마찰이 낮아야 한다 (가입 없음, 링크만으로 시작).

타겟 축소에 따른 이점: 상황 분기(커플/룸메이트/가족)가 사라져 온보딩이 한 단계 짧아지고, 카피·브랜딩을 신혼 맥락으로 고정할 수 있으며, v2에서 신혼부부 특례대출·특별공급 정보 연계라는 명확한 확장 경로가 생긴다.

## 4. 핵심 컨셉과 조율 모델

조율 방식은 **필수/선호 분리 필터링**을 채택한다 (가중치 병합 방식 기각 사유: 한쪽의 강한 조건이 평균에 희석됨).

조건 5개는 두 유형으로 나뉜다.

**상한 입력형 (항상 적용)** — 통근시간, 예산. 각자 상한 값을 입력하며 분류 대상이 아니다. 통근시간은 두 사람의 상한을 각각 적용하고, 예산은 낮은 쪽을 기본 적용하되 상한이 다르면 충돌로 표시한다.

**분류형 (필수/선호/무관)** — 평형, 년식, 인프라. 각자 3단계로 분류하며, 필수는 인당 최대 2개로 제한한다 (분류형 3개 전부를 필수로 지정하면 교집합 0건 위험이 커지므로. DB 트리거로 강제).

매칭 파이프라인:

1. **1차 필터 — 상한 + 필수 교집합.** 두 사람의 통근 상한과 합의 예산을 만족하고, 한쪽이라도 필수로 지정한 분류형 조건을 모두 충족하는 구역만 후보로 남긴다.
2. **2차 랭킹 — 선호 가중합.** 남은 후보를 A·B의 선호(nice) 충족 수로 점수화해 정렬한다.
3. **충돌 리포트.** 자동 절충하지 않는다. "A는 필수, B는 무관"인 조건과 상한이 다른 연속 조건(예산)을 명시하고, 각 조건을 완화했을 때 후보가 몇 곳 늘어나는지 수치로 제시한다. 결정은 두 사람이 한다.
4. **동의 무게 차등화.** 모든 공동 행동에 사전 동의를 걸면 마찰이 과도해지므로, 행동의 되돌림 가능성에 따라 동의 방식을 3단계로 나눈다.
   - **구역 제외 (즉시 + 투명)**: 한쪽이 결과 화면에서 구역을 제외하면 즉시 적용된다. 제외된 구역은 삭제되지 않고 "제외한 구역" 그룹으로 별도 묶여 제공되며, 누가 제외했는지 표시되고 **두 사람 모두 언제든 복구(undo)할 수 있다**. 신뢰는 사전 허락이 아니라 투명성과 가역성으로 지킨다.
   - **기준 변경 (제안-동의)**: 후보 세트 전체를 바꾸는 규칙 변경이므로 한쪽의 제안 → 상대의 동의를 거쳐야만 적용된다. 일방 변경 불가.
   - **확정 (양측 확정 = 공동 확정)**: 결과 화면 상단에 나와 상대의 확정 여부를 보여주는 플로팅 상태 표시를 상시 노출한다. 각자 "확정" 버튼을 누르면 본인 확정으로 반영되며, **클릭과 동시에 확정 구역 리스트 이미지가 기기에 저장**된다 (별도 다운로드 단계 없음). 두 사람 모두 확정한 시점에 공동 확정이 성립한다. 별도 승인 화면·모달 없이 상태 표시만으로 진행된다.
   - **확정 후 변경 규칙**: 한쪽만 확정한 상태에서 후보 리스트를 바꾸는 행동(제외·복구·기준 변경)이 발생하면 기존 확정은 자동 해제되고 당사자에게 표시된다. 두 사람의 확정 이미지가 서로 다른 리스트를 담는 상황을 방지하기 위함이다.

시스템의 역할 경계: 알고리즘은 트레이드오프 정보를 제공하고, 절충의 판단은 사용자에게 남긴다. "B가 양보하세요"류의 처방적 메시지는 금지한다.

**실시간 동기화와 부재중 변경 전달**: 두 사람이 동시 접속 중이면 제외·확정·제안이 Supabase Realtime으로 상대 화면에 즉시 반영되고, presence로 "상대가 지금 보고 있어요"를 표시한다. 앱 밖 푸시는 v1에서 제공하지 않는 대신(iOS 웹푸시 제약, 알림톡 비용), 재방문 시 상단에 부재중 변경 요약("A가 다녀갔어요: 수잔2동 제외 · 확정 대기 중")을 배너로 제공한다. 상대의 확정을 기다리는 상태에서는 카카오톡 리마인드 링크 보내기를 수동 액션으로 제공한다.

## 5. 조건 목록

가는 집과 동일한 원칙 — 공공 API로 검증 가능한 조건만 채택한다.

| # | 조건 | 유형 | 기준 정의 | 데이터 소스 |
|---|---|---|---|---|
| 1 | 통근시간 | 상한 입력 · 개인별 거점 | 문에서 문까지, 도보·환승 대기 포함 | ODsay (대중교통), v1은 대중교통만 |
| 2 | 예산 | 상한 입력 · 낮은 쪽 적용 | 최근 6개월 실거래 평균 대비 | 국토부 실거래가 API |
| 3 | 평형 | 분류형 | 전용면적 구간 (예: 59㎡ 이상) | 실거래가 API |
| 4 | 년식 | 분류형 | 준공 10년 이내 = 신축 | 건축물대장 |
| 5 | 인프라 | 분류형 | 마트·병원·공원 접근성 통합 지수 | 상가정보 + 심평원 + 도시공원 API |

인프라는 마트(차량 10분)·병원(차량 20분)·공원(도보 10분) 세 항목의 충족 수로 산출하는 통합 지수다. v1에서는 3항목 중 2개 이상 충족 시 "인프라 충족"으로 판정하고, 세부 가중치 조정은 v2로 미룬다.

제외된 조건과 사유: 역세권(통근시간 조건이 실질적으로 포섭 — 역이 가까우면 통근시간에 반영됨), 대단지(신혼부부 의사결정에서 우선순위 낮음, 조건 수 축소), 학군(가족 타겟 제외로 불필요), 소음·쾌적함·개발 호재(측정 불가), 치안(구역 단위 점수화의 오해석 리스크).

## 6. 사용자 플로우와 화면

플로우: 거점·통근 상한 → 예산 상한 → 조건 분류 → 초대/대기 → 결과 → 조율 (6화면). 상황 선택 화면은 신혼부부 단일 타겟 확정으로 삭제.

| 화면 | 핵심 설계 결정 |
|---|---|
| ① 거점·통근 | "직장"이 아닌 "자주 가는 곳"(직장/학교/부모님 집/직접 입력). 통근 상한 슬라이더가 지도 도달 범위와 연동. "문에서 문까지, 도보·환승 대기 포함" 기준 명시 |
| ② 예산 상한 | 슬라이더 입력. 두 사람의 상한이 다르면 낮은 쪽을 기본 적용하고 충돌로 표시 |
| ③ 조건 분류 | 분류형 3개(평형·년식·인프라)를 탭 방식 카드 소팅. 필수 슬롯 2개를 상시 노출해 "아껴 쓰는 자원"으로 인식시킴. 분류 완료 조건은 하단 칩으로 누적, 재분류 가능 |
| ④ 초대/대기 | A 입력 완료 후 초대 (빈손 초대 대비 B 완료율 우위 가정). 카카오톡 공유 + 동적 OG 태그 ("OO님이 신혼집 찾기에 초대했어요"). "상대 입력 완료 전 내 조건 비공개" 명시 (입력 편향 방지, RLS로 강제) |
| ⑤ 결과 | 헤드라인이 추천 근거를 먼저 제시. 구역 카드마다 A/B 통근시간 병기. 카드별 제외 버튼 — 제외 시 즉시 적용되고 "제외한 구역" 그룹으로 이동, 제외자 표시 + 양측 복구 가능. 상단 플로팅으로 나/상대 확정 여부 상시 표시, "확정" 클릭 시 본인 확정 반영 + 구역 리스트 이미지 즉시 저장, 양측 확정 시 공동 확정 성립. 재방문 시 부재중 변경 요약 배너. 결과 공유 진입점 배치 |
| ⑥ 조율 | 조건 조정 시 후보 수 실시간 반영. 이산 충돌은 양자택일 버튼, 연속 충돌(예산)은 두 사람 값을 양 끝으로 하는 슬라이더. 변경은 제안 → 상대 동의로만 적용 |

**결과 공유 (v1 포함)**: 결과 화면에서 추천 구역 요약을 이미지 카드로 생성해 카카오톡·인스타그램으로 공유한다. 카드에는 구역명, 매칭률, 두 사람의 통근시간을 담되 예산 등 민감 정보는 기본 제외한다 (포함 여부 토글 제공). 공유 카드에 서비스 유입 링크를 포함해 신규 A 획득 채널을 겸한다 — 신혼부부는 준비 과정을 지인·SNS에 공유하는 경향이 강해, 결과 공유가 자연스러운 획득 루프가 될 수 있다.

## 7. 지표 체계

| 구분 | 지표 | 정의 |
|---|---|---|
| North Star | 확정 버튼 클릭률 | 공동 결과 도달 세션 중 확정 클릭(1인 이상)이 발생한 세션 비율. 비율 지표는 규모를 담지 못하므로 주간 확정 세션 수(볼륨)를 반드시 병행 모니터링한다 |
| H1 핵심 | 공동 후보 확정률 | 공동 결과 도달 세션 중 7일 내 **양측 모두 확정 클릭**(공동 확정)이 성립한 비율. `result_shared`는 보조 신호 |
| H1 선언 | 결정 자기보고 분포 | 피드백 배너 1단계 응답 분포: 정해졌어요 / 아직 고민 중 / 여기엔 없어요 |
| KPI① | B 초대 수락률 | 초대 링크 클릭 → B 입력 시작 (H2) |
| KPI② | B 입력 완료율 | B 입력 시작 → 조건 분류 완료 (H2) |
| 진단 | 조율 진입률 × 제안 수락률 | 매트릭스로 해석: 진입高×수락高 = 건강한 협상 / 진입高×수락低 = 조건 설계 마찰 / 진입低×확정高 = 첫 결과가 충분 / 진입低×확정低 = 가치 전달 실패 |
| 보조 | 결과 공유율 / 공유 경유 신규 세션 | 획득 루프 관찰용 (공유는 확정 신호에도 포함되므로 이중 역할) |
| 가드레일① | 필수 교집합 0건 비율 | 완결 세션 중 후보 0곳 비율. 상승 시 조건 설계·필수 제한 재검토 신호 |
| 가드레일② | A 온보딩 이탈률 | 거점 입력 진입 → 초대 화면 도달 실패 비율 |

**지표 설계 원칙**

행동 신호와 선언 신호를 교차 검증한다. 확정 신호가 발생했는데 자기보고가 "여기엔 없어요"인 세션 비율이 높으면 행동 신호의 정의가 잘못됐다는 뜻이므로, 이 불일치율 자체를 분기마다 점검한다.

조율 진입률은 방향이 양가적(깊은 사용일 수도, 첫 결과 불만족일 수도)이므로 가설 검증 지표가 아닌 진단 지표로만 쓴다.

`b_completed` → 첫 확정 신호까지의 소요 시간 분포는 기술 통계로 관찰하되, 대조군이 없으므로 "결정이 빨라졌다"는 인과 주장에는 사용하지 않는다.

이벤트 트래킹은 Mixpanel, 세션 단위 funnel: `session_created → invite_sent → invite_opened → b_started → b_completed → result_viewed_a → result_viewed_b → area_excluded → area_restored → result_shared → proposal_created → proposal_accepted → result_confirmed(role 프로퍼티로 개인 클릭 기록, 공동 확정은 양측 발화로 도출)`.

## 8. MVP 범위

**포함**: 6화면 전체 플로우, 상한+필수 필터링·선호 랭킹 엔진, 통근시간 계산(대중교통), 충돌 리포트, 제안-동의 조율 루프, 구역 소프트 제외(그룹화·양측 복구), 확정 플로팅 + 양측 저장에 의한 공동 확정(구역 리스트 저장 포함), Realtime 동기화 + 재방문 변경 요약, 카카오톡 초대, 결과 이미지 카드 공유, 구역 단위 추천 (행정동 프리셋).

**초기 지원 지역 (확정)**: 강남·판교 두 업무지구의 통근권이 겹치는 경기 동남부 — 성남시, 하남시, 과천시, 의왕시, 용인시(수지구·기흥구), 광주시를 1차 범위로 한다 (행정동 약 60~80개). 선정 근거: ① "한 명은 강남, 한 명은 판교" 조합이 흔해 통근 균형이라는 핵심 기능이 가장 잘 작동하는 무대, ② 서울 대비 신혼부부 예산과 맞는 진입 가격대의 구역이 충분, ③ 행정동 규모가 통근 캐시·데이터 배치 비용으로 감당 가능. 확장 기준: NSM이 안정적으로 발생하고 교집합 0건 비율(가드레일①)이 임계 이하로 유지되면, 인접 통근권(분당선·신분당선·경강선 연장 축) 순으로 넓힌다. 지역 확장은 areas 테이블 시드 추가만으로 가능하도록 지역 하드코딩을 금지한다.

**제외 (v1 아웃오브스코프)**:
- 매물 단위 추천 — 구역까지만 추천하고 매물 탐색은 기존 부동산 앱에 위임. 매물 데이터 크롤링은 ToS 위반으로 채택 불가 (가는 집에서 확인된 제약)
- 자차 통근시간 — 대중교통만 v1 제공
- 신혼부부 특례대출·특별공급 정보 연계 (v2 확장 경로)
- 인프라 세부 가중치 커스터마이즈
- 계정 시스템 고도화 — B는 익명 인증으로 시작, 계정 전환은 v2

**폴백 요구사항**: 필수 교집합이 0건일 때 "A의 필수만 반영한 후보"와 "B의 필수만 반영한 후보"를 나란히 제시한다. 빈 결과 화면은 세션 폐기로 직결되므로 v1 필수 기능이다.

**피드백 수집 (v1 포함)**: 결과 화면 하단에 낮은 강도의 배너/시트로 노출한다. 1단계는 결정 자기보고 질문 — "가볼 구역이 정해졌나요?" [정해졌어요 / 아직 고민 중 / 여기엔 없어요] — 으로, H1 선언 지표의 측정 도구를 겸한다. "여기엔 없어요" 선택 시에만 짧은 서술형 입력을 이어서 받는 2단계 구조로 한다.

트리거 조건: 주 트리거는 `result_viewed_b` 이후 결과 화면 체류(스크롤 또는 일정 시간)가 확인된 시점, 보조 트리거는 `proposal_accepted` 시점이다. 둘 중 먼저 도달하는 시점에 노출하고, 세션당 1회만 노출한다 (재노출 금지).

배제 원칙: 온보딩 전 화면(거점·예산·조건 분류)과 B의 초기 진입 직후에는 절대 노출하지 않는다. B 퍼널이 가드레일②로 관리되는 최대 리스크이므로, 피드백 수집이 이탈 요인이 되어서는 안 된다.

| 백엔드 | Supabase | Anonymous Auth (B 무가입 참여), Realtime (B 완료 실시간 반영), RLS (조건 비공개 정책 DB 강제) |

| 배포 | Vercel | — |
| 외부 API | 카카오 로컬 (주소 검색), ODsay (통근), 국토부 실거래가, 카카오톡 공유 SDK | — |

아키텍처 제약: 통근시간 계산은 `(출발 거점 100m 격자, 구역, 수단)` 단위로 캐시하며, 구역은 행정동 프리셋 대표 좌표로 고정한다. DB 스키마 및 RLS 정책은 schema.sql(v0.2, PRD v0.4 기준 반영 완료) 참조. **PRD v0.6 반영을 위한 스키마 v0.3 변경 필요 항목**: ① `area_exclusions` 테이블(session_id, area_code, excluded_by, restored_at — 소프트 제외/복구 이력), ② `confirmations` 테이블(session_id, participant_id, area_codes text[], criteria_snapshot jsonb, price_snapshot jsonb, confirmed_at — 개인 확정 기록, 세션당 2행이 모이면 공동 확정), ③ Realtime publication에 두 테이블 추가.

확정 시 저장되는 구역 리스트는 파일이 아니라 구조화 데이터(구역 코드 배열 + 확정 시점의 기준·시세 스냅샷 jsonb)로만 서버에 보관하고, 이미지·텍스트 파일은 다운로드 시점에 클라이언트에서 생성한다 (서버 저장 없음).

**확정 산출물 사양**: 사용자는 두 형식으로 내려받는다. ① PNG 이미지 1장 — 확정 구역 카드(구역명·확정 시점 시세·A/B 통근시간)를 세로로 이어붙인 공유 카드 형태로, 결과 화면 카드 컴포넌트를 재사용해 렌더링한다. ② 텍스트(.txt) — 부동산 전달·프린트 용도를 고려해 마크다운 기호 없는 순수 텍스트로 조립한다. 두 형식 모두 동일한 확정 스냅샷(confirmations)에서 생성한다. 생성 경로는 이미지가 DOM → 캔버스 → PNG(html-to-image류), 텍스트가 문자열 조립 → Blob 다운로드이며, 체감 소요는 이미지 기준 1초 안팎, 텍스트는 즉시다.

**산출물 구현 주의사항**: ① 웹폰트 사용 시 폰트 로드 완료를 기다린 뒤 캡처를 시작한다 (미완료 상태 캡처 시 폰트 깨짐 발생). ② 저사양 기기·확정 카드 수 증가로 렌더링이 길어질 수 있으므로, 로딩 스피너는 0.3초 이상 걸릴 때만 표시한다. ③ 모바일에서는 파일 다운로드 대신 OS 공유 시트(Web Share API)로 연결해 사진 저장·카카오톡 전송으로 이어지게 한다. ④ 확정 클릭과 이미지 저장이 한 동작으로 묶이므로, iOS Safari의 사용자 제스처(user activation) 유효 시간 안에 공유 시트가 떠야 한다 — 렌더링(약 1초)이 비동기로 길어지면 제스처가 만료되어 공유 시트가 차단될 수 있으니, 결과 화면 진입 시 이미지를 미리 렌더링해두고 확정 클릭 시엔 완성본을 즉시 넘기는 사전 렌더링 방식을 기본으로 한다. ⑤ 텍스트(.txt) 형식은 확정 완료 후 별도 버튼으로 유지한다 (자동 저장은 이미지만).

## 10. 리스크와 열린 질문

| 리스크 | 내용 | 대응 |
|---|---|---|
| B 퍼널 붕괴 | B가 링크를 열지 않거나 중도 이탈하면 서비스가 성립하지 않음 | H2 퍼널을 최우선 계측. 초대 메시지 문구·OG 이미지 실험 |
| 교집합 0건 빈발 | 상한 2개 + 필수 조건 조합이 배타적이면 빈 결과 반복 | 필수 2개 제한 + 폴백 화면. 가드레일① 모니터링 |
| 통근 API 쿼터 | ODsay 무료 쿼터 초과 | 격자 캐시 + 행정동 프리셋. 초기 구역 수 제한 |
| 데이터 신선도 | 실거래가 평균이 시장 급변기에 왜곡 | 최근 6개월 집계 + 갱신 시점 표기 |
| 조건 축소의 부작용 | 조건 5개가 신혼부부 의사결정을 충분히 커버하지 못할 가능성 (예: 역세권을 통근시간이 대체하지 못하는 케이스) | 결과 화면에 "빠진 조건이 있나요?" 피드백 수집, 조건 추가는 데이터 기반으로 결정 |

열린 질문: ① 공유 카드의 민감 정보(예산) 기본 노출 정책 최종 확정.

**의도적 보류 사항**: 저장 구역 시세 변동 알림 등 재방문 유도 기능은 v1에서 의도적으로 제외한다. 사용자가 실제로 무엇을 저장·추적하고 싶어 하는지에 대한 데이터가 아직 없는 상태에서 기능을 확정하면 추측 기반 설계가 되므로, 우선 피드백 수집(8장)과 기본 지표로 재방문 동기를 관찰한 뒤 방향을 정한다.
